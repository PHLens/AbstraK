# P0.1 Provider Conformance

P0.1 verifies the smallest reproducible boundary between AbstraK and a model
API. It does not implement an agent loop, parse kernel code, execute tools, or
compare model quality.

## Acceptance boundary

A provider/model pair is ready for the controlled pilot only when it has:

- strict provider and exact model manifests;
- one non-streaming candidate per logical request;
- no fallback, cache, implicit retry, alias map, or dropped parameters;
- a returned text response, finish reason, model identity, and usage record;
- a valid plain-JSON action containing the probe nonce;
- a private artifact bundle with no credential material.

Every report separates three decisions:

- `transport_ready`: the endpoint returned one normalizable response with complete core usage;
- `action_protocol_ready`: strict JSON and the system-only nonce both survived;
- `pilot_ready`: both checks pass, the model identity is exact, and the source tree is clean.

A mutable alias can therefore pass endpoint conformance while remaining
ineligible for the controlled pilot.

LiteLLM is used only as the API transport. `request.transport.json` records the
arguments handed to the SDK, not its final HTTP body. AbstraK records the SDK
response object with `capture_fidelity: sdk_object`; this is not a byte-exact
HTTP capture.
Native tool calling is deliberately deferred because providers implement it
differently and the controlled track uses a common textual action protocol.

## Offline validation

```bash
uv run abstrak-provider validate \
  --provider path/to/provider.yaml \
  --model path/to/model.yaml
```

This checks schemas, cross-references, retry/fallback policy, and emits stable
manifest hashes. It does not resolve credentials or use the network.

## Live smoke probe

Set only the environment variables named by the provider manifest, then run:

```bash
uv run abstrak-provider smoke --live \
  --provider path/to/provider.yaml \
  --model path/to/model.yaml
```

The command performs exactly one potentially billable request. It creates an
ignored mode-0700 run directory under `artifacts/provider-conformance/`, then
seals successful terminal bundles read-only with mode 0500. Each bundle contains:

- `manifest.resolved.json`;
- `request.logical.json` and `request.transport.json`;
- `response.sdk.json` and `response.normalized.json`, or `response.error.json`;
- append-only `events.jsonl`;
- `result.json` and `sha256sums.txt`.

No production provider/model manifest should be committed until its exact API
model identifier, endpoint behavior, and artifact have been reviewed. Rolling
aliases must use `model_id_policy: mutable_alias`; snapshot-like identifiers
must use `exact` and declare `expected_returned_model`.

The model manifest must also set `allow_live_probe: true`. This and the CLI's
`--live` flag are separate safeguards against accidentally executing the
non-runnable example manifests.

Remote base URLs must use HTTPS and cannot contain credentials, query strings,
or fragments. Plain HTTP is accepted only for loopback conformance tests.

## Exit codes

- `0`: all required checks pass;
- `2`: invocation, manifest, or secret preflight failure;
- `3`: transport/provider failure;
- `4`: response or textual action contract failure;
- `5`: artifact or harness failure.

## Deferred from P0.1

Streaming, native function calling, Responses API, concurrency, automatic
retry, long-context behavior, cost estimation, Agent memory, GPU execution,
and workload evaluation belong to later milestones.

Local read-only sealing prevents accidental mutation within the harness. Paper
artifacts still require an externally anchored checksum or immutable object
store; local file permissions alone are not a provenance guarantee.
